Eu protocolo, tu protocolas, nós protocolamos.

Não é de hoje que APIs existem.

Uma API (Application Programming Interface) é um conjunto de rotinas e padrões estabelecidos por um software para a utilização das suas funcionalidades por aplicativos que não pretendem envolver-se em detalhes da implementação do software, mas apenas usar seus serviços.

Eu protocolava, tu protocolaras, nós protocoláramos.

Muito antes desse frenezi todo de webapps e aplicativos mobiles que estamos vivendo protocolos de API já existiam.

  • SOAP

  • REST

Um protocolo para todos dominar

Desenvolvimento web não é trivial, são muitas tecnologias e assuntos para dominar.
Uma delas que esta presente na vida de todos é o HTTP.
Entre seus muitos detalhes se destacam os verbos:

  • GET
  • POST
  • PUT
  • DELETE
  • OPTIONS

REST Representational State Transfer

Em português Transferência de Estado Representacional
REST é uma abstração do HTTP para "padronizar" o acesso a recursos/dados.

  • Não guarda estado (cookie, session, etc)
  • Tira proveito dos verbos HTTP
    • get para recuperar
    • post para criar
    • put para atualizar
    • delete para apagar

RESTful e outros termos

RESTful é o nome que damos a um sistema que segue os principios do REST

Recurso é a informação (comumente uma entidade)

Endpoint é o nome dado a URI única que permite acesso a um recurso

Client quem esta acessando o recurso (navegador, apps, IoT, desktop, etc)

Observação importante:
nem todo client tem a capacidade/facilidade de enviar/armazenar cookies/session esse é um dos motivos que APIs REST não usam cookies e afins.

Recursos e Previsibilidade

Uma boa API deve ser previsivel usando os verbos http corretamente.

GET /users
Recuperar uma coleção de registros do recurso
POST /users
Criar novo registro no recurso
GET /users/73
Recuperar um registro específico do recurso
PUT /users/73
Atualizar um registro específico do recurso
DELETE /users/73
Remover um registro específico do recurso

Recuperar uma coleção de registros de um sub-recurso
POST /users/73/questions
Criar novo registro no sub-recurso

Respostas

Um endpoint pode variar o tipo de arquivo que ela responde, porém a "estrutura" não varia.

  • JSON
  • XML
  • CVS
  • YAML
  • HTML
  • HAL
  • JSON-API
  • OData
  • Json-LD
  • Json-Schema
  • GraphQL

Bate-Bola / Jogo Rápido

  • Um recurso SEMPRE deve responder da mesma estrutura.
  • Não tente "economizar" respostas.
  • Sua resposta é o mesmo que você recebe.
  • Evite expor seu "banco", use transformers.
  • Crie uma padrão para erros.
  • Versione.
  • Pagine.
  • Forneça metadados.
  • Códigos HTTP existem para serem usados.
  • Não use sessão.
